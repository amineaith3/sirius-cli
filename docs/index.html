Sirius-CLI Documentation

Technical reference manual for the Sirius rapid prototyping engine.

1. Introduction

Sirius-CLI is a rapid application development tool that infers schemas from raw data files (CSV, Excel, SQLite, JSON) and compiles them into a complete, production-ready, containerized full-stack application.

Unlike standard boilerplate generators, Sirius-CLI parses your data relationships, handles English pluralization rules automatically, wires up SQL foreign keys, and generates an enterprise-grade React data grid complete with server-side pagination, sorting, search, and CSV/Excel exports.

2. Installation

Sirius-CLI is distributed via PyPI. We recommend installing it globally or within a dedicated virtual environment.

Prerequisites

  • Python 3.9+ (Required for the CLI tool)
  • Node.js 16+ (Required for running the generated frontend locally)
  • Docker (Optional, for deploying the generated docker-compose.yml)
pip install sirius-cli

Verify the installation:

sirius-init --version

3. Core Concepts

3.1 Schema Inference

When you pass a CSV or Excel file to Sirius, the engine uses pandas to read the headers and infer standard data types (Strings, Integers, Booleans, Floats). It sanitizes column names into safe SQL identifiers.

3.2 Relationship Mapping

Sirius-CLI features an intelligent relationship resolver powered by the inflect library. If the engine detects a column ending in _id (e.g., customer_id), it will:

  1. Extract the base name (customer).
  2. Resolve the exact English plural (customers).
  3. Check if a customers table exists in the current parsing context.
  4. If found, it automatically wires a SQLAlchemy Foreign Key and generates React UI dropdowns linking the two tables.

This happens automatically without requiring manual configuration files.

3.3 Dynamic Form Validation

Generated React CRUD forms are strictly validated based on inferred data constraints natively via HTML5:

  • Required Fields: Reads NOT NULL constraints via PRAGMA table_info (SQLite) or infers isnull() heuristics (Pandas) to automatically inject required attributes.
  • Numeric Bounds: Extracts min and max extremes from CSV/Excel data and sets them on numeric inputs.
  • Email Regex: Automatically tags column names containing "email" with type="email".

3.4 Server-Side Search & Sorting

Sirius-CLI does not rely on fragile client-side filtering. Instead:

  • Search: A debounced (350ms) search input fires a ?search= query. The FastAPI backend queries the database using standard wildcard string matching across all text fields.
  • Sorting: Clicking table headers toggles a server-side ?order_by= and ?dir= API request.

3.5 Relationship Navigation

Generated applications feel like fully fleshed-out enterprise admins rather than raw data grids:

  • Label Resolution: Foreign Key IDs (e.g., User #4) are transparently resolved to human-readable names (e.g., alice@example.com or Alice) using zero-cost state lookups.
  • Highlight Navigation: Clicking a Foreign Key badge instantly routes the user to the parent record (?highlight=ID). The target row is automatically scrolled into view and flashes with a brand-colored pulse animation to anchor the user's context.

3.6 Data Exporting

Every generated data table features one-click bulk exports directly to standard formats. The FastAPI backend streams the entire current dataset matching active search and sort parameters directly to the browser as either a .csv or binary .xlsx (Excel) file.

4. CLI Reference

The CLI exposes two primary commands: init and update.

4.1 sirius-init init

Used to scaffold a completely new project.

sirius-init init <project_name> [OPTIONS]
Flag Description
--csv <path> Path to input CSV file. Repeatable for multiple files.
--excel <path> Path to input Excel (.xlsx) file. Repeatable.
--db <path> Path to an existing SQLite database to introspect.
--config, -c Path to a JSON configuration defining explicit schema.
--theme, -t Tailwind theme color. Defaults to blue.
--port, -p Backend port (default 8000). Updates Docker and .env files.
--api-url Overrides the frontend VITE_API_URL. Default points to localhost:port.
--pg, --mysql Generates the project with PostgreSQL or MySQL drivers/pools instead of SQLite.
--auth Generates JWT Bearer authentication, Login UI, and protects API routes.
--admin-user Default admin username (default: admin). Requires --auth.
--admin-pass Default admin password. If omitted, a secure password is auto-generated.
--no-seed Skips inserting rows from source files into the generated SQLite DB.
--from-url URL of a public CSV/JSON/Excel file to fetch and scaffold from (repeatable). Downloads are cached locally.

4.2 sirius-init update

Used to inject new tables or columns into an existing project. It safely regenerates models, schemas, and UI, and automatically runs an Alembic autogenerate migration.

sirius-init update <project_path> [OPTIONS]

Accepts the same input flags (`--csv`, `--db`, `--from-url`, etc.) as the init command, plus `-m` for custom migration messages.

4.3 sirius-init preview

Provides an instant, ephemeral way to visualize your inferred data schemas and UI without writing a single file to disk. The engine creates an in-memory SQLite database, constructs a dynamic FastAPI router on the fly, and serves a beautiful Vue 3 + Tailwind CSS frontend interface directly from the CLI.

sirius-init preview [OPTIONS]

Accepts `--csv`, `--excel`, `--db`, and `--from-url` flags for data input, plus `--port` to specify the local server port.

5. Architecture Deep Dive

Sirius-CLI does not hide its generated code behind abstraction layers. It outputs raw, standard Python and TypeScript files that you own and can edit freely.

5.1 Backend (FastAPI)

  • main.py: The core FastAPI router. Automatically generates fully functional CRUD endpoints for every table. Includes server-side ?search= and ?order_by= handlers, and /export endpoints generating CSVs and Excel files.
  • database.py: Configures the SQLAlchemy SessionLocal pool. For Postgres/MySQL, it enforces reading the connection string from the DATABASE_URL environment variable.
  • models.py: Generates strict SQLAlchemy declarative base models with correct foreign key mappings and indexes.
  • schemas.py: Generates Pydantic v2 models (Create, Update, Response) for strict request validation and serialization.

5.2 Frontend (React + Vite)

  • TableCrud.tsx: A lean orchestration component generated per-table. It handles data fetching, sorting, searching, and state management, then delegates all rendering to the src/components/ library.
  • src/components/SiriusTable: A fully dynamic data grid driven by a runtime ColumnSchema[] JSON array. Renders all column types (Boolean, DateTime, FK badges) without any hard-coded Jinja loops in the table body.
  • src/components/SiriusPagination: A standalone cursor-based pagination footer.
  • src/components/SiriusBadge: A reusable boolean status indicator (TRUE/FALSE).
  • src/components/SiriusDropdown: A decoupled FK <select> resolver used in Create/Update modals.
  • src/components/SiriusError: A unified server connectivity error display.
  • Dashboard.tsx: Integrates Recharts to provide a dynamic overview of table row counts and schema structure.

5.3 Migrations (Alembic)

During scaffolding, Sirius initializes Alembic in the backend directory. It writes the `env.py` to target your specific database, runs alembic revision --autogenerate to snapshot the initial models, and executes alembic upgrade head to create the physical tables.

6. Security & Authentication

JWT Implementation

When the --auth flag is provided, Sirius generates an auth.py file containing OAuth2 Password Bearer logic via python-jose and passlib. All standard endpoints in main.py are subsequently injected with a Depends(get_current_user) dependency.

Production Checklist

Important: Generated projects are scaffolds and must be secured before deploying to production.
  • SECRET_KEY: The JWT token generation relies on a SECRET_KEY environment variable. If omitted, the server generates a random key on startup, which will invalidate all user sessions upon server restart.
  • Database Credentials: If you generated a Postgres or MySQL project, the database.py file contains placeholder credentials. You must pass a valid DATABASE_URL via environment variables.
  • CORS Origins: The generated main.py restricts CORS to `localhost:5173` and the explicitly configured `--api-url`. Ensure your production frontend domain is added to _allowed_origins.
